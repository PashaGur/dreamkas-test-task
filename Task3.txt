Чек-лист для проверки функционала, который отправляет заявку в сервисный центр
Перебираем возможные варианты для положительного тестирования)

- Заголовок должен отображаться на каждой странице.
- Сервис выбирается (из предложенного списка/вводится вручную)
- Указываются обязательные данные пользователя (e-mail для связи, id оборудования (по которому идет запрос в сервис-центр))
наличие звездочки у всех обязательных полей
- Заполняется поле описания проблемы (также обязательное поле, заявка не должна поступать в сервис "пустой" или с пустыми обязательными полями)
- Необязательные поля (заполняются случайно / оставляются пустыми)
Проверка действительности вводных данных, соответствия введенных данных допустипых для полей значениям
Параметры форм, в которых возможно удаление или любая другая модицикация данных
- Отправляется заявка в сервис центр.
Для первичного положительного тестирования ключевыми факторами будет являться сохранность данных, их формата и возможность оставить заявку в центр
(письма поступают в сервис-центр)
На случай необходимости срочной поддержки / сохранения возможности обработки обращений при сбоях, на странице оформления заявки должен быть указан телефон
для обращения в техподдержку по срочным вопросам.
Также, может быть включен всплывающий бот-помошник по обработке заявок (помогающий ответить на частые вопросы / дублирующий возможности заполения через форму).

Отрицательные тесты
Должны отсутствовать ссылки, ведущие к одной странице
Если ли странцицы, на которые не указана ссылка
Отсутсвтие нерабочих ссылок
Сохраняются ли введенные данные при обновлении страницы? проверка в включенными и выключенными cookies
Проверка сообщения об ошибке в случае ввода неверных данных, неверного формата, пустых обязательных полей, уже введенные данные должны остаться,
 ошибочное поле может быть "стерто".
Убедиться в наличии шифрования cookies перед записью на ПК пользователя
Числовые поля не должны принимать буквы, в этом случае долны отображаться соотв. сообщения об ошибках.
Отрицательные значения в числовых полях, если они разрешены
Максимальная длина поля описания ошибки (невозможность ввода символов сверх дозволенного или оповещение, когда пользователь вводит больше, т.е. писать возможно,
 отправлена будет лишь дозволенная часть).

1. Выбор сервисного центра
Для удобства пользователя может использоваться функция определения геолокации, которая поможет автоматически определить ближайший к владельцу сервис.
* Кроссплатформенная проверка, а также проверка в разных версиях браузера
(синтаксические ошибки HTML, доступность сайса для поисковых сервисов, наличие точной карты сайта в форматах XML и HTML)
- Проверка отправки-получения данных о геолокации, формата данных
- Проверка функционала при медленном/среднем/быстром интернете, при больших нагрузках на сервис по приему данных.
- Возможность указать адрес на карте.
- Возможность указать ближайшую станцию метро, всплывающие подсказки ближайших сервисов
- В случае, если сервисов много и для удобства реализован скроллинг по тексту/выпадающий список/переключение по вкладкам (1-n) , проверить кликабельность вариантов
 (будет ли автоподстановка адреса сервиса в заявку качестве выбора) , покажется ли точка сервиса и точка пользователя на карте после данного выбора
 (если такие имеются)
Панель скролла должна появляться только тогда, когда она требуется
Данные больших списков должны быть расположены в хронологическом порядке

2. Срочная поддержка
Подразумевается, что заявку можно оформить без авторизации (тестирование авторизации к функционалу отправки заявок не относится), отличие может быть
в автоподстановке полей авторизованного пользователя, где стоит проверить формат данных, и их соответствие полям.
- За пределами формы заявки должны быть указаны номера / почта / адреса сервисных центров.
Если они на отдельных страницах, каждая должна быть доступна, должны отображаться все данные (возможно кликабельная почта для связи с конкретным сервисом,
отображение сервисов на карте.

3. Сценарии для гарантийного и постгарантийного обращения, отличия.
Гарантийный случай подразумевает бесплатную диагностику (если нет, то проверка функционала вкючения стоимости)
Весь период обращения сервис ограничен по срокам, следовательно, нужна проверка функционала (возможности заполнения, автоматического расчета,
 взаимосвязанности данных) по расчету примерного времени, строго от дня обращения до окончания периода диагностики, до окончания периода ремонта.,
  далее оповещение о готовности. (проверка данного функ.)

В постгарантийном случае после диагностики идет проверка функционала оповещения
после диагностики, возможности отказаться, оплатить онлайн/наличными в сервисе
Проверка повторного уведомления, формы уведомления, таймера.

4. Доп проверки
Попытка отравить несколько идентичных запросов
С несуществующего e-mail, который написан в верном формате
Попытка скопировать текст описания проблемы в поле описания (другой шрифт, цвет, отступы).
Попытка отправить запросы с разных адресов, но с идентичными данными оборудования (с учетом того, что каждое оборудование имеет уникальные данные,
 которые повторяться не могут)
Страницы не должны содержать орфографических ошибок.
